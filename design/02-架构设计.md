# 架构设计文档

## 「AI超能力鉴定 × OPC工作台」技术架构

**原则**：极简、可维护、5天交付、手机优先

---

## 一、技术选型

### 前端

| 层面 | 选型 | 理由 |
|------|------|------|
| 框架 | **原生 HTML/CSS/JS**（无框架） | 零构建、零依赖、体积极小、秒加载 |
| 样式 | 原生 CSS + CSS Variables | 移动端适配、主题一键切换 |
| 路由 | Hash-based SPA（手写50行） | 无框架依赖，页面间平滑过渡 |
| 动画 | CSS transition + 少量 JS | 页面切入、卡片翻转、加载动效 |

### 后端

| 层面 | 选型 | 理由 |
|------|------|------|
| 运行时 | **Node.js**（原生 http 模块） | 极简，无框架依赖 |
| API对接 | MiniMax API (`/v1/text/chatcompletion_v2`) | 国内直连、中文能力强 |
| 提示词管理 | 独立的 `prompts/` 目录，每个场景一个 `.md` | 非技术人员也能改 |

### 部署

| 层面 | 选型 | 理由 |
|------|------|------|
| 托管 | **Vercel** | 免费、国内可访问、自动HTTPS、一键部署 |
| 域名 | Vercel 自动分配 `*.vercel.app` | 无需另购域名 |

## 四、数据流 & 状态管理

```
全局状态（app.js 维护）：

{
  route: '/',                    // 当前页面
  quiz: {
    currentQuestion: 0,          // 当前题号（0-5）
    answers: [null×6],           // 每题的选项序号
    personality: null,           // 计算结果：人格类型
  },
  workbench: {
    scenario: null,              // 选中的场景ID
    inputs: {                    // 用户输入
      industry: '',
      sellingPoint: '',
      audience: '',
    },
    result: null,                // AI生成结果
    loading: false,
  }
}
```

**数据流动**：
```
用户操作 → 更新内存状态 → 渲染UI → 调用API → 更新状态 → 重新渲染
```

无后端数据库，无登录，无持久化——状态仅在浏览器内存中，刷新即清空。符合线下快闪场景。

## 五、API接口定义

### 接口1：测评人格分析

```
POST /api/quiz
Content-Type: application/json

Request:
{
  "answers": [1, 0, 2, 3, 1, 0]  // 6道题的选项序号(0-3)
}

Response 200:
{
  "personality": "content_alchemist",
  "title": "内容炼金术师",
  "icon": "🔥",
  "description": "你天生擅长把模糊的想法变成让人停不下来的文字...",
  "tags": ["文案写作", "短视频脚本", "小红书运营"],
  "recommendScenario": "script"
}
```

**实现**：后端根据答案映射规则计算人格（不需要调AI，纯逻辑），若用户要求用AI生成更个性化描述，再调用MiniMax。

### 接口2：场景内容生成

```
POST /api/generate
Content-Type: application/json

Request:
{
  "scenario": "script",           // script | poster | sop | ip
  "industry": "知识付费",         // 用户输入：行业
  "sellingPoint": "AI写作课程",   // 用户输入：核心卖点
  "audience": "职场新人"           // 用户输入：目标受众
}

Response 200:
{
  "content": "🎬 短视频口播脚本\n【钩子】...（完整内容）",
  "scenario": "script"
}
```

**实现**：后端读取对应场景的 Prompt 模板，填入用户输入，调用 MiniMax API 生成。

### 错误响应（通用）

```
Response 500:
{
  "error": "生成失败，请稍后重试",
  "fallback": "（预制的兜底样例内容）"
}
```

## 六、关键设计决策

### 为什么不用框架？
- 项目体量极小（5个页面、3个API接口）
- 零构建步骤 → 改完即部署，5天交付节奏友好
- 体积极小（<50KB）→ 活动现场4G秒加载
- 无框架依赖 → 不担心版本兼容

### 为什么测评不调AI？
- 6道单选题的映射逻辑，纯代码规则即可，延迟为0
- 避免"等AI思考"带来的体验断裂
- 如需个性化描述，可在结果页异步加载AI润色版本

### 为什么生成结果不存服务器？
- 线下场景，无需用户系统
- 用户手机截图就是"存档"
- 极简架构，零数据库运维
